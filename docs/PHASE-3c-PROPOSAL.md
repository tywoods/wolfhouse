## Executive summary

**Phase 3c** moves the **WhatsApp guest booking assistant** from **Airtable-primary** to **Postgres-first**, with **Airtable as a temporary mirror** only where staff UI still needs visible rows (payment link, status, conversation flags). Long-term intent is **not** to deepen Airtable dependency — remove mirror nodes when a proper staff/guest UI reads Postgres.

The implementation target is the **local Stripe fork** of Main, generated by [`scripts/build-main-local-stripe.js`](../scripts/build-main-local-stripe.js), not the hosted 167-node export. Today that fork has **189 nodes**, **64 Airtable** nodes, and **one** Postgres node (`Ensure Booking In Postgres`) on the **Stripe checkout** path only. 3c expands Postgres to **hold creation**, **availability**, **conversation/message state**, and **existing-booking** paths while preserving Phase **2f** resolver behavior and Phase **2c** payment wiring.

**Recommended substeps:** **3c.a** inventory → **3c.b** read-only PG availability → **3c.c** PG booking hold/upsert → **3c.d** conversation/message plan → **3c.e** fork generation with PG gates → **3c.f** payment/confirmation contract check → **3c.g** E2E local sign-off.

### 1.4 Architecture principles (approved direction)

3. **Not Airtable source of truth** — PG failure **blocks** Airtable mirror writes (same rule as [`PHASE-3-PROPOSAL.md`](PHASE-3-PROPOSAL.md) dual-write).  
4. **No long-term Airtable breadcrumbs** — document mirror nodes as **deprecated**; remove when internal UI exists.  
5. **Local/staging only** until extended sign-off and Phase 4 plan.

## 3. Desired target architecture

### 3.1 Write order (dual-write window)

For each guest-facing mutation:

```text
1. Validate (resolver, overlap, idempotency)
2. WRITE Postgres (transaction where possible: booking + related rows)
3. IF PG ok → mirror Airtable (booking, conversation, message, beds only if needed)
4. IF PG fail → NO Airtable write; log automation_errors
5. IF PG ok, AT fail → log; retry sync from PG id (do not roll back money)
```

### 3.2 Authority matrix (target)

| Entity | Phase 2 today | 3c target (local MVP) | Phase 4 target |
|--------|---------------|------------------------|----------------|
| `bookings` | AT create (Main); PG on Stripe path | **PG first** + optional AT mirror | PG |
| `booking_beds` | 3b forks; Main inline AT only | Main uses **PG overlap**; assign via 3b webhook or shared SQL | PG |
| `conversations` | AT | PG first + optional AT mirror | PG |
| `messages` | AT | PG first + optional AT mirror | PG |
| `payments` / `payment_events` | PG (2b) | **Unchanged** | PG |
| Payment link on booking | AT column | Mirror from PG/checkout URL | PG or staff UI |

### 3.3 Stripe / Send Confirmation (frozen contracts)

| Contract | 3c must preserve |
|----------|------------------|
| **Ensure Booking** returns `booking_id` | Yes — extend to all hold paths, not only Stripe branch |
| **`booking_code`** | `WH-` + Airtable record id pattern until unified code generation |
| **`send_confirmation` flag** | Set only by Stripe webhook — Main must not clear it incorrectly |
| **Send Confirmation** | Reads PG; 3c may mirror `status=confirmed` to AT **after** 2d (optional substage) |
| **`__NULL__` / `NULLIF`** | All new Postgres nodes in Main fork |

### 3.4 Environment / feature flags (proposed)

Optional per workflow (document in runbook when implemented):

```text
MAIN_DATA_SOURCE=postgres | dual | airtable_compat
```

- **postgres** — PG only, no AT mirror (dev strict mode)  
- **dual** — default for 3c MVP where mirror needed  
- **airtable_compat** — emergency rollback (local only)

## 5. Safety rules

| Rule | Detail |
|------|--------|
| **No payment writes in Main** | No INSERT/UPDATE/DELETE on `payments` or `payment_events`; no changing `payment_status` except via existing 2b webhook |
| **No hosted export edits** | `n8n/Wolfhouse Booking Assistant  - Main.json` read-only |
| **No active path collisions** | Only local Stripe fork on `booking-assistant` |
| **Generated JSON only** | Logic in `build-main-local-stripe.js` + `scripts/lib/*` |
| **PG failure blocks AT mirror** | No orphan Airtable bookings without PG row |
| **Idempotency** | Upsert on `booking_code`; Stripe/webhook idempotency unchanged |
| **Active hold protection** | Single active hold per phone/window — extend 2f rules with PG queries |
| **Rollback** | Deactivate 3c fork; re-import hosted Main; `npm run db:sync` for beds; delete fixture bookings by code |
| **Never** | Force-push; prod AT automation toggles; activate on Cloud without approval |
