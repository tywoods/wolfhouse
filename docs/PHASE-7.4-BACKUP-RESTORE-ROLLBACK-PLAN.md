## 1. Objective

- **No pilot/staging operation without verified backups and a tested restore drill.**
- Every migration must have a rollback or forward-fix strategy documented before it is applied.
- Workflows and client config must be recoverable without re-engineering.
- Restore must be tested (not just configured) before real guest messages are handled.

## 4. Restore drill (required before any pilot)

The restore drill must be **executed and documented** before the first live guest message is handled. It is a go/no-go gate (§10).

### Drill procedure

1. **Take a snapshot** of the staging app DB at a known state (or use the most recent automated backup). Record timestamp and row counts.
2. **Restore** the snapshot into an **isolated restore DB** (separate server/DB; never overwrite the source).
3. **Verify table counts** match the snapshot baseline:
   - `bookings`, `payments`, `payment_events`, `booking_beds`, `conversations`, `staff_handoffs` — compare row counts.
4. **Verify sample rows** (read-only SELECT): pick 2–3 representative bookings, 1 payment, 1 handoff — confirm key fields are intact.
5. **Verify app connectivity**: point the staff API (in a read-only test mode or a local instance) at the restore DB; confirm `GET /staff/query` returns the expected row counts.
6. **Verify n8n workflows**: export all staging workflows, delete from a test n8n instance, re-import — confirm all workflows are present and inactive.
7. **Document time-to-restore**: record wall-clock time from snapshot trigger to verified connection.
8. **Record outcome**: pass/fail for each step; sign off with dev/ops owner.

### Drill gate

- All 7 steps above must pass before any pilot window is approved.
- Drill result is documented in a dated entry appended to this file (§4.1 below) or a linked runbook.

## 5. Migration rollback strategy

All existing and planned migrations are catalogued below. Each must have a rollback/forward-fix strategy documented **before** it is applied to staging or production.

### Classification

- **Additive:** adds tables, columns, indexes, or constraints. No data dropped. Rollback is `DROP TABLE / DROP COLUMN IF EXISTS` (safe in staging pre-pilot; risky in production if rows exist).
- **Destructive / renaming:** drops or renames tables/columns; requires data migration. Never auto-run in production without backup + explicit approval.

### Migration catalogue

| Migration | File | Type | Status | Rollback strategy |
|---|---|---|---|---|
| **001** `init` | `001_init.sql` | Additive (schema init) | Applied to local dev | Forward-fix only in production (never drop core tables). Staging pre-pilot: drop + re-init if needed. |
| **002** `package_pricing` | `002_package_pricing.sql` | Additive (columns/tables) | Applied to local dev | `DROP TABLE IF EXISTS package_pricing` or similar; safe pre-pilot in staging. |
| **003** `rename_hostel_to_client` | `003_rename_hostel_to_client.sql` | **Destructive rename** | Applied to local dev | Cannot safely reverse in production once data exists. Pre-pilot staging: restore from backup + re-init. After pilot: forward-fix only. |
| **004** `payment_schema_phase2` | `004_payment_schema_phase2.sql` | Additive | Applied to local dev | Drop added tables/columns if no data. Production: forward-fix (disable feature flags). |
| **005** `payment_intent_unique_partial` | `005_payment_intent_unique_partial.sql` | Additive (index only) | Applied to local dev | `DROP INDEX IF EXISTS` — low risk, no data dropped. |
| **006** `confirmation_sent_at` | `006_confirmation_sent_at.sql` | Additive (column) | Applied to local dev | `ALTER TABLE bookings DROP COLUMN IF EXISTS confirmation_sent_at` — safe if no downstream dependency. |
| **007** `add_addon_orders` | `007_add_addon_orders.sql` | Additive (new tables) | Applied to local dev | Drop added tables if pre-pilot and no data. Production: disable feature flag for add-ons; forward-fix. |
| **008** `add_staff_handoffs` | `008_add_staff_handoffs.sql` | Additive (new tables) | Applied to local dev | Drop `staff_handoffs` / `staff_tasks` / indexes if pre-pilot and no data. Production: disable `STAFF_ACTIONS_ENABLED`; do not drop if handoffs exist. |
| **009** `auth_staff_users` (planned) | `009_auth_staff_users.sql` (not created) | Additive (new tables) | Designed only — not created | Drop `staff_users` / `auth_sessions` if pre-pilot and no users exist. Production: disable auth middleware; forward-fix. |

### Rules for applying migrations

1. **Backup before apply** — snapshot the target DB before running any migration. No exceptions.
2. **Apply to staging first** — never run a new migration in production without staging proof.
3. **Smoke after apply** — run a read-only verification (table exists, row counts stable, no execution errors) before resuming app operations.
4. **Rollback/forward-fix plan confirmed** — the rollback strategy for the migration must be documented in this table before it is applied.
5. **No auto-run destructive migrations** — any migration that drops data or renames columns in production requires explicit written approval and a separate rollback data-safety review.
6. **Production rollback default: forward-fix** — in production with live data, prefer disabling the feature flag and adding a compensating migration over dropping tables that contain data.

## 6. Workflow rollback

| Rule | Detail |
|---|---|
| **Export before import** | Before importing a new or updated workflow into staging n8n, export the **current version** of all workflows and store (repo or dated folder). |
| **Import inactive** | All imported workflows start inactive. No auto-activation on import. |
| **One active per window** | At most one workflow active at any time during a planned test window. Deactivate after. |
| **Rollback = deactivate + re-import** | If a newly activated workflow causes problems: deactivate it immediately, then re-import the previous version from the stored export, verify it is inactive, and document the incident. |
| **Version tracking** | Each workflow stored in the repo (`n8n/phase2/*.json`) is tied to a git commit hash. When activating a staging workflow, record the commit hash in the test window log. |
| **Do not reuse activation windows** | If a test window reveals issues, do not resume activation without a documented rollback assessment. |

## 7. Feature-flag emergency toggles

These are the **fast levers** to stop dangerous operations without a deployment. They must be documented, understood, and drillable by the ops owner before any pilot.

| Lever | Mechanism | Effect | Who can flip |
|---|---|---|---|
| Stop all WhatsApp sends | `WHATSAPP_DRY_RUN=true` (redeploy or env update) | All WhatsApp send nodes log and skip Graph API call; no real messages | Dev/ops |
| Disable staff write endpoint | `STAFF_ACTIONS_ENABLED=false` (redeploy or env update) | `POST /staff/handoff/:id/resolve` returns 403; no DB writes | Dev/ops |
| Disable live payment link sending | Deactivate `Create Payment Session` n8n workflow | No new Stripe checkout sessions created | Dev/ops (n8n UI) |
| Disable confirmation send | Deactivate `Send Confirmation` n8n workflow | No confirmation messages or `booking.confirmed` updates | Dev/ops (n8n UI) |
| Disable Stripe webhook processing | Deactivate `Stripe Webhook Handler` n8n workflow | Webhooks queue but are not processed; Stripe will retry | Dev/ops (n8n UI) |
| Disable main assistant | Deactivate `Wolfhouse Booking Assistant - Main` n8n workflow | No bot responses to incoming WhatsApp messages | Dev/ops (n8n UI) |
| Disable add-on writes | Feature flag (to be defined when add-on write surface exists) | No add-on fulfillment writes | Dev/ops |
| Human takeover | Deactivate all bot workflows; staff handle conversations directly via WhatsApp | Full human control | Dev/ops + owner |

**Note:** flipping env vars requires a container redeploy on Azure Container Apps (fast, seconds). Deactivating n8n workflows is instant from the n8n UI. Both are valid emergency actions.

## 8. Incident / rollback runbook (high-level)

This is the decision tree for a live incident. A detailed, executed runbook is a pre-pilot deliverable (Stage 7.6).

1. **Detect** — alert fired (monitoring, §Stage 7.5) or staff reports unexpected behaviour.
2. **Stop live sends** — flip `WHATSAPP_DRY_RUN=true` or deactivate Main workflow (§7). No new guest messages sent.
3. **Stop payment links** — deactivate Create Payment Session workflow (§7).
4. **Freeze staff writes** — set `STAFF_ACTIONS_ENABLED=false` (§7).
5. **Snapshot DB** — take an on-demand backup of the staging/production app DB immediately (before any remediation).
6. **Notify staff** — Cami and Ale informed; agreed comms channel (WhatsApp group or equivalent). All in-flight guest conversations go to human handling.
7. **Diagnose** — inspect n8n execution logs, audit log, DB counts. Identify root cause.
8. **Restore or forward-fix** — if data corruption: restore from the pre-incident snapshot into an isolated DB, verify, promote with approval. If logic error: forward-fix via migration + flag change.
9. **Verify counts** — after any restore or fix, verify protected table counts (`bookings`, `payments`, `payment_events`, `booking_beds`) match the pre-incident baseline.
10. **Resume** — only after step 9 passes and the ops owner (Cami/Ale) approves. Re-enable in reverse order of step 2–4.
11. **Post-incident** — document cause, fix, duration, impact; update this plan if gaps found.
